# אופטימיזציה של עמוד ההצגות (V2)

## הבעיה

כל כניסה לעמוד "כל ההצגות" לקחה כ-2 שניות.
הסיבה: בכל ניווט לעמוד, הושלחו 6-7 שאילתות לבסיס הנתונים — בלי שום caching.

## מה עשינו

### 1. שמירת נתונים מחושבים על מודל ההצגה (Denormalization)

**לפני:** בכל טעינת עמוד, חישבנו ממוצע דירוג וספירת ביקורות באמצעות `JOIN` ו-`GROUP BY` על טבלת הביקורות — שאילתת SQL כבדה.

**אחרי:** הוספנו שני עמודות לטבלת `Show`:

- `avgRating` — ממוצע הדירוג
- `reviewCount` — מספר הביקורות

העמודות מתעדכנות אוטומטית בכל יצירה, עריכה או מחיקה של ביקורת.
כך מיון לפי דירוג הפך משאילתה מורכבת עם `JOIN` לפשוט `ORDER BY avgRating`.

### 2. Cache לתצוגת ברירת מחדל

**לפני:** כל כניסה לעמוד ההצגות (בלי סינון) הפעילה את כל השאילתות מחדש — גם אם הנתונים לא השתנו.

**אחרי:** התצוגה ללא סינון (העמוד הראשון, מיון ברירת מחדל) נשמרת ב-cache ל-120 שניות.
כניסה חוזרת תוך דקותיים מוגשת מיידית מה-cache — בלי לגעת בבסיס הנתונים.

### 3. אינדקסים בבסיס הנתונים

הוספנו אינדקסים לטבלת `Show`:

- `@@index([theatre])` — מאיץ סינון לפי תיאטרון ואת dropdown התיאטרונות
- `@@index([avgRating])` — מאיץ מיון לפי דירוג

### 4. טעינה מועדפת לתמונות מעל ל-fold

**לפני:** כל 12 התמונות של הכרטיסיות נטענו ב-lazy loading — גם אלה שנראות מיד.

**אחרי:** 4 התמונות הראשונות מקבלות `priority`, כך הדפדפן מתחיל לטעון אותן מיד ולא מחכה שהן ייכנסו ל-viewport.

## תוצאה

| מדד                      | לפני     | אחרי            |
| ------------------------ | -------- | --------------- |
| שאילתות DB (תצוגה רגילה) | 6-7      | 0 (cache)       |
| שאילתות DB (עם סינון)    | 6-7      | 2-3             |
| זמן תגובה (תצוגה רגילה)  | ~2 שניות | מיידי (מ-cache) |
